MONA (Multi-Omics Network Analysis)

Languages: Python, R
Rocks HPC, Linux, NumPy, SciPy, Bioinformatics

Project Description

During my year-long internship at Moffitt Cancer Center, I was lucky to be working in the Bioinformatics Core on one of their new research programs, MONA. MONA was originally conceived for the purpose of finding new interactions between genes and metabolites associated with lung cancer, but the flexible approach lends it toward anything involving genes, metabolites, proteins, micro RNA, etc.

Aside from its network-based approach for integrative data analysis, it has a customizable feature for using weighted correlations to add speculative interactions between expressors (genes and metabolites) which aren't currently annotated.

The end goal is to give researchers a way to identify likely interactions to do further study on.

Interactive Network Visualization

For the data visualization, I used the statistics-focused R language. R was a natural choice because of how easily it could handle structured data and the availability of the Shiny framework. RShiny is useful for quickly creating interactive web applications that incorporate R functionality.

I experimented with several different visualization packages before settling on the visNetwork package. Since it created graphs that updated in real-time and could be pulled around by the user, it was a natural choice for large-scale data, and fun to use.

MONA incorporates a network-based approach for integrative data analysis, using empirical correlations to make edges with expressors that aren’t previously annotated.

Genetic Algorithm Environment

Language: C++
Visual Studio 2015

Project Description

I often have ideas for ways to apply genetic algorithms, but usually end up not implementing them due to the time required to get them working. GAB-Class is designed to make it as fast as possible to get genetic algorithms working in C++.

The project includes a GeneticAlgorithm class which holds the data and runs each generation, a Chromosome abstract class that provides common methods, and two static libraries for crossover and mutation methods.

With this setup, it only takes roughly 15 minutes to create a new genetic algorithm and get started writing the fitness function.

Basic Keyframed Animation

One of the most popular animation techniques for hobbyist game devs is using keyframed animations to block out every action. In the figure below, the dots on the left represent bone rotations at certain times during a looping jog animation. These are called keyframes.

Keyframed animations offer great control over specific poses during actions. In a run cycle, it is easy to define the reaching/passing points. However, it tends to create stiff animations where the actor looks like they are moving to the beat of a metronome.

Adding intermediate keyframes and shifting the timing of keyframes can help mitigate this, but it is a time-consuming process and makes it difficult to modify the animations later. I often found myself tweaking part of an animation to make it smoother, only to find that it ruined the animation 6 frames later.

Word Search Game for Amblyopia Therapy

Language: JavaScript
Unity3D, Amblyopia research

What is Amblyopia?

Ambylopia is an eye problem where a person has a weakened eye. In many cases, the person favors their strongest eye, and the weak eye can begin to wander. This is commonly known as having a "lazy eye."

According to Lazyeyegames.com, almost 3% of people have this problem. One of my family members has it, and I decided to recreate her favorite word search game with a color-scheme designed to help her eye if she used red-blue glasses while playing it.

According to my main reference, Lazyeyegames.com, amblyopia involves the brain ignoring a certain amount of input from the weak eye. In order to improve, the brain must be forced to use both eyes simultaneously. This point affected the game design and steered me away from making a game that simply "exercised" the weak eye, and toward a game design that required the player to use both eyes at the same time.

To accomplish this, I decided to create a handful of minigames using the idea behind a popular word search game. Being a game where the player searches for words among random letters, the players’ eyes must be watching many things simultaneously and looking for patterns.

How is the game set up to use both eyes?

The player wears red-blue glasses. The game colors are tweaked so that the red aligns with the red lens and the blue with the blue lens. When letters in these colors are drawn on a black screen, the player sees a different image in each eye. The game is set up so that letters are randomly assigned to be blue or red.

Through experimentation, I found that viewing other colors through the glasses was disorienting and stressful. This was due to the colors looking very different in each eye, and being able to see both colors simultaneously. However, yellow and purple were fairly neutral, so I used those to convey information to the player quickly.

For example, the score was shown in a neutral color so that the player could quickly glance at it before returning to the red/blue letters. Also, the amount of time left each round was indicated by a large box that shrank in size. Giving it a neutral color and a basic shape made it easier to see in peripheral vision.

Implementation

To handle so many words in a quick manner, I loaded 235,886 words into a dictionary. Since the data did not include plurals, the game automatically assumes that valid words with an “S” at the end are also valid. Furthermore, if a word is rejected, the player can press a key to have it added to a secondary dictionary of custom words.

Game modes

I added 8 game modes total. Here is a sampling:




EXPLORATION:

The player starts with only the word "CAT" on the top left. Typing a word causes all the letters next to that word to be discovered. The goal is to uncover the entire grid before time runs out.



LAVA ESCAPE:

Letters sink into lava, according to how “heavy” they are. Typing words adds new random letters around those words, but also increases the weight. The player attempts to stay above the lava for as long as possible.



ZOMBIES:

Typing a word holds the zombies at bay, but if no word is typed within a couple seconds, the zombies are given 1 turn to spread. Eventually, they take over the entire grid.



WOLVES:

This mode increases the grid size, shrinks the letters, and adds random blank spaces. To increase the tension, there is a sound loop of wolves howling, so that players can feel like they're about to be eaten if they don't find enough words.


Making it fun and responsive!

It is difficult for people with amblyopia to use their eyes together and sometimes they will get headaches while doing so. To make the game less stressful, I included settings for letter size and the speed of the game so players could adjust the game to their comfort level.

I also wanted to make the game as enjoyable as possible, so people would do their therapy regularly. This was fairly challenging because I couldn't use techniques like screenshake and particle bursts. The visuals needed to be simple and extremely clear at all times. In the end, I decided to add these features:

  • Letters briefly expand when selected, with a satisfying sound
  • Successfully finding a word triggers a happy coin sound
  • Cheesy announcers congratulate the player when they find large words (these audio clips were from a package purchased on the Unity Asset Store)
  • The game keeps record of the top 10 high scores in all game modes, as well as the total number of words found

    Music Evolution

    Language: VBA (I wasn't a computer science major yet)

    Project Description

    This project served as my senior project while getting a math degree. I only knew VB6 at the time and decided to use VBA in Excel. I had it output the music to an Excel sheet mimicing a piano roll, which I would then play on my MIDI keyboard.

    That took a very long time (I play guitar, not piano), so for most of the development, I wasn't able to actually hear the music.

    Preparation Phase

    The program started by generating a pseudo-random chord progression in C Major. This didn't have a direct effect on the notes, but influenced the fitness function.

    The 2nd preparation phase was deciding where notes would occur for each instrument. I wrote a GA for this

    I separated these phases from the main GA because it would let my laptop not die when performing calculations, and it allowed me to keep the same chord progressions and note locations across several iterations of the GA. This gave me a clearer picture of how the GA was developing.

    This is why some of the samples have such a similar sound - they were evolving based on the same chord progressions and/or note locations.

    Evolving the Notes

    This was the most important and processor-heavy part of the project. The GA started by filling in the previously-assigned note locations with completely random pitches in the ranges of each instrument. It then created 99 copies so there were 100 total quartet passages, or "chromosomes" (in GA terms).

    Selection Phase

    Each generation, the chromosomes were scored according to a fitness function meant to reflect how "good" they should sound.

    Research has shown that popular songs tend to flow the way the brain expects them to, but also have a small, important level of unexpectedness. I used that principle by recording how much variance there was in each chromosome and combining that with a score of how well it fit music theory and the intended chord progression.

    This was done in several passes:

    NOTE LEVEL:

    Each note was evaluated based on how it flowed into and out of the notes around it (based on intervals), and the harmonies it created with the notes the other instruments were playing. It also accounted for the time since those other notes were started - playing notes simultaneously made their harmony score/penalty more pronounced than one note playing over other notes being held.

    MEASURE LEVEL:

    The notes in each measure were evaluated based on factors such as how well they represented the chord progression, and whether "unexpected" notes were in proper locations (ie. notes that would act as "leading tones" into a chord change).

    SONG LEVEL:

    I used statistical analysis to encourage chromosomes with interesting distributions. This helped solve a few problems. Withit this factor, chromosomes were overusing "safe" notes and sticking to the same intervals.

    All this was taken into account for the final score.

    Copy Phase

    Normally there is a Crossover Phase, where low scoring chromosomes are recreated from crossed-over data from high scoring ones. I don't remember where I learned what a GA was, but I didn't know about crossovers.

    So I just copied the top scoring chromosome directly to all the others. It was essentially a GA set up with extreme elitism and low diversity.

    Mutation Phase

    Each generation, all but the top scoring chromosome were mutated randomly. The mutations ranged from "take the 47th cello note and move it down 3 semitones" to "grab 6 notes and pull them up/down a measure."

    I found that the mutations had a huge influence on how long the GA had to run before getting a good result. For example: A melody could suddenly jump 2 octaves higher, stay there for a few notes, and then return to the normal range. This is obviously bad, but anytime a single note would randomly move downward, it would make the chromosome even worse. This is why I used so many types of mutations - including ones that would move entire sections of neighboring notes.

    First Results

    Things started out pretty randomly... The algorithm had a long ways to go.



    After improving the algorithm, this flighty tune appeared, before quickly descending into madness.



    Eventually, the music started sounding like... something you would call 'music'.



    Best Result

    Finally, I got a piece that had some interesting melodies and countermelodies! It sounds like it really wants to build up to something great - before getting slightly confused.




    FUTURE PLANS

    I plan to translate the project to C# and take advantage of having a faster PC than I did back in 2012, as well as my increased knowledge of genetic algorithms. The obvious improvement is to incorporate crossovers.

    I'd also like to experiment with detecting when results become stagnant, and splitting the GA population for a set number of generations. Elite individuals will apply simulated annealing in an attempt to converge on a better solution while non-elite (I like to call them "volatile") individuals have an increased mutation rate and switch to using rank selection.

    (Non-GA translation) This would give the best individuals a chance to further prove themselves, while giving other solutions a chance. At least, it would be fun to try!
